데이터베이스(DB) 기초

1. DB란 무엇인가?

데이터베이스(Database)는 여러 사람이 공유하여 사용할 목적으로 통합하여 관리되는 데이터의 집합입니다.

  • 중복성 최소화: 데이터가 여기저기 흩어져 있는 것을 방지
  • 무결성 유지: 데이터의 정확성과 일관성 보장
  • DBMS: 이런 DB를 관리하는 시스템 (예: MySQL, Oracle)

DB의 기본 아키텍처

우리가 쓰는 앱이나 웹사이트는 직접 데이터를 조작하지 않고, 서버를 통해 DBMS에 데이터를 요청합니다.

2. RDBMS (관계형 DB)

Relational Database Management System

데이터를 행(Row)과 열(Column)로 이루어진 테이블(표) 형태로 저장하고, 테이블 간의 관계를 연결하는 가장 표준적인 DB입니다.

a. 테이블 (Table) 예시

엑셀의 시트(Sheet)와 매우 유사한 구조입니다.

고객ID (Primary Key) 이름 (Column) 나이 (Column) 등급 (Column)
1 홍길동 (Row) 25 VIP
2 김철수 (Row) 30 Gold
  • Primary Key (기본키): '고객ID'처럼 각 사람을 절대 헷갈리지 않게 구분하는 고유 번호입니다.

b. SQL (구조화된 질의어) 예시

DB에게 "이것 좀 해줘"라고 명령하는 언어입니다.

1. 테이블 만들기 (DDL)


CREATE TABLE 고객 (
  고객ID INT PRIMARY KEY,
  이름 VARCHAR(50),
  나이 INT
);

2. 데이터 넣기 (DML)


INSERT INTO 고객 (고객ID, 이름, 나이) 
VALUES (1, '홍길동', 25);

3. 데이터 찾기 (DML)


SELECT 이름, 나이 FROM 고객 WHERE 나이 >= 25;
-- 결과: 홍길동(25), 김철수(30)

c. 정규화 (Normalization) 예시

데이터의 중복을 막기 위해 테이블을 의미에 맞게 쪼개는 과정입니다.

❌ 정규화 전 (중복 발생: 홍길동의 정보가 계속 반복됨)

주문번호고객명연락처주문상품
101홍길동010-1111노트북
102홍길동010-1111마우스

✅ 정규화 후 (고객 테이블과 주문 테이블로 분리, '고객ID'로 연결)

고객ID (PK)고객명연락처
1홍길동010-1111
주문번호고객ID (FK)주문상품
1011노트북
1021마우스

d. ERD (Entity-Relationship Diagram)

테이블 간의 관계를 설계도로 그린 것입니다.

  • 1:1 관계: 사람 - 주민등록증
  • 1:N 관계 (가장 흔함): 고객 1명이 여러 번(N) 주문을 함
  • N:M 관계: 학생(N)과 수강과목(M) (보통 중간 테이블을 둬서 1:N 두 개로 풉니다)

e. 트랜잭션과 ACID

은행 계좌 이체를 생각해보세요. 내 돈은 빠져나갔는데, 상대방 계좌엔 안 들어간다면?

  • A (원자성): 이체는 완전히 성공하거나, 아예 시도 전으로 돌아가야 합니다.
  • C (일관성): 이체 전후로 두 사람의 잔고 합은 같아야 합니다.
  • I (고립성): 동시에 여러 명이 이체를 해도 서로 꼬이지 않아야 합니다.
  • D (지속성): 이체 성공 메시지가 떴다면, 컴터가 꺼져도 기록은 남아야 합니다.

3. NoSQL (비관계형 DB)

정해진 표(Table) 형식을 벗어나, 데이터 특성에 맞게 자유롭고 빠르게 저장합니다.

a. 문서형 (Document)

데이터를 중첩된 JSON 형태로 저장합니다. 스키마가 자유롭습니다.


{
  "이름": "홍길동",
  "취미": ["독서", "게임"],
  "주소": { "도시": "서울", "구": "강남구" }
}
  • 대표 DB: MongoDB

b. Key-Value & 와이드 컬럼

  • Key-Value: '학번:1234' -> '홍길동' 처럼 단순한 키-값 쌍. 매우 빠름. (예: Redis)
  • 와이드 컬럼: 행(Row)마다 컬럼 개수와 종류가 달라도 허용됨. 대용량 데이터 분산 처리에 특화. (예: Cassandra)

c. 그래프 (Graph)

데이터 자체보다 '데이터 간의 관계'를 탐색할 때 씁니다.
(예: "A의 친구의 친구가 좋아하는 영화 추천")

  • 대표 DB: Neo4j

4. 특수 목적 DB

a. 시계열 DB & 인메모리 DB

  • 시계열 DB: 주식 차트, 온도 센서 등 '시간'에 따라 쏟아지는 데이터를 1초 단위로 기록하고 조회할 때 씁니다. (예: InfluxDB)
  • 인메모리 DB: 하드디스크가 아닌 RAM에 데이터를 저장해 압도적으로 빠릅니다. (예: Redis) 주로 데이터를 잠깐 임시 저장(캐싱)할 때 씁니다.

b. 벡터 DB (Vector DB)

ChatGPT 같은 AI 기술의 핵심입니다. 텍스트나 이미지를 숫자 배열(벡터)로 변환해 저장하고, 숫자의 거리가 가까운 '유사한' 데이터를 순식간에 찾아냅니다.

5. 클라우드 DB란?

AWS, GCP 같은 서비스가 제공하는 "빌려 쓰는 데이터베이스" 입니다.

왜 클라우드 DB를 쓸까요?

  • 설치 제로: 서버 사고, 선 연결하고, 프로그램 설치할 필요가 없습니다.
  • 관리 자동화: 매일 밤 백업, 해킹 방어, 고장 시 복구를 알아서 해줍니다.
  • 무한 확장: 트래픽이 몰리면 버튼 한 번으로 DB 성능을 2배, 10배로 늘릴 수 있습니다.

결론

세상에 모든 걸 완벽하게 처리하는 만능 DB는 없습니다.

돈이 걸린 결제 시스템처럼 무결성이 생명이면 RDBMS를,
수백만 명의 접속을 감당하고 유연해야 하면 NoSQL을,
AI 검색이 필요하면 벡터 DB를 섞어 쓰는 것이 현대의 아키텍처입니다.
