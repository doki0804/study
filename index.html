데이터베이스(DB)에 대하여

1. DB란 무엇인가?

데이터베이스(Database)는 여러 사람이 공유하여 사용할 목적으로 통합하여 관리되는 데이터의 집합입니다.

  • 중복성 최소화: 엑셀 파일처럼 여기저기 흩어져 있는 것을 방지
  • 무결성 유지: 데이터의 정확성과 일관성 보장
  • DBMS: 이런 DB를 관리하는 소프트웨어 (MySQL, Oracle 등)

DB의 기본 아키텍처

우리가 쓰는 앱은 직접 데이터를 조작하지 않고, 서버를 통해 DBMS에 데이터를 요청하여 가져옵니다.

2. RDBMS (관계형 DB)

데이터를 행(Row)과 열(Column)로 이루어진 테이블(표) 형태로 저장하고, 테이블 간의 관계를 연결하는 가장 강력하고 표준적인 DB입니다.

a. 테이블 (Table) 구조

엑셀의 시트(Sheet)와 매우 유사한 형태를 가집니다.

고객ID (Primary Key) 이름 (Column) 나이 (Column) 등급 (Column)
1 홍길동 (Row) 25 VIP
2 김철수 (Row) 30 Gold
  • Column (열): 데이터의 속성 (이름, 나이 등)
  • Row (행): 실제 데이터 한 건 (레코드)

b. 기본키(PK)와 외래키(FK)

테이블 간의 관계(Relationship)를 맺어주는 핵심 장치입니다.

[고객 테이블]

고객ID (PK)이름
1홍길동
2김철수

[주문 테이블]

주문번호 (PK)주문자ID (FK)상품명
1011노트북
1021마우스
  • Primary Key (기본키): 각 행을 고유하게 식별 (주민번호 같은 역할)
  • Foreign Key (외래키): 다른 테이블의 PK를 참조. 부모 테이블에 없는 값은 넣을 수 없음 (참조 무결성 보장).

c. SQL 기초 (DDL, DML, DCL)

RDBMS와 대화하기 위한 언어입니다.

1. 테이블 생성 (DDL)


CREATE TABLE 고객 (
  고객ID INT PRIMARY KEY,
  이름 VARCHAR(50)
);

2. 데이터 삽입/조회 (DML)


INSERT INTO 고객 VALUES (1, '홍길동');
SELECT * FROM 고객;

d. JOIN (테이블 합치기)

나눠진 테이블들을 관계(PK-FK)를 기준으로 합쳐서 조회합니다.


SELECT 고객.이름, 주문.상품명 
FROM 고객 
INNER JOIN 주문 ON 고객.고객ID = 주문.주문자ID;

결과: 홍길동-노트북, 홍길동-마우스

  • INNER JOIN: 양쪽 테이블에 모두 조건이 맞는 데이터만 조회 (교집합)
  • LEFT JOIN: 왼쪽 테이블 기준, 오른쪽은 매칭되는 것만 붙여줌 (없으면 NULL)

e. 트랜잭션과 ACID

데이터 무결성을 지키는 논리적 작업 단위입니다. (예: 계좌 이체)

  • A (Atomicity, 원자성): 모두 성공하거나 아예 실패해야 함 (All or Nothing).
  • C (Consistency, 일관성): 작업 전후로 규칙(잔고 마이너스 불가 등)이 유지되어야 함.
  • I (Isolation, 고립성): 동시에 실행되는 트랜잭션은 서로 영향을 주지 않음.
  • D (Durability, 지속성): 성공한 결과는 컴퓨터가 꺼져도 영구 보존됨.

f. Index (인덱스)

데이터 검색 속도를 엄청나게 높여주는 책의 목차(찾아보기)입니다.

  • 장점: 책을 처음부터 끝까지 다 읽는 것(Full Table Scan)을 막고, SELECT 속도를 비약적으로 향상시킵니다.
  • 단점: 데이터가 추가/수정/삭제 될 때마다 목차도 갱신해야 하므로 쓰기 속도가 느려지고 추가 용량을 차지합니다.

g. 데드락 (Deadlock)

트랜잭션들은 데이터를 수정할 때 Lock(잠금)을 겁니다.

  • 교착 상태: 두 트랜잭션이 서로 상대방이 잡고 있는 Lock이 풀리기만을 무한정 기다리는 상태입니다. (좁은 골목길 마주 본 자동차)
  • 해결: DBMS는 데드락을 감지하면, 하나를 강제로 취소(Rollback)시켜 길을 터줍니다.

h. 정규화 (Normalization)

데이터의 중복을 막고 이상현상(Anomaly)을 방지하기 위해 테이블을 의미에 맞게 쪼개는 설계 과정입니다.

  • 제1정규형: 모든 컬럼의 값은 원자값(단일값)을 가져야 함
  • 제2정규형: 기본키의 일부에만 종속되는 컬럼 분리 (부분 함수 종속 제거)
  • 제3정규형: 일반 컬럼끼리 종속되는 것 분리 (이행적 함수 종속 제거)

*주의: 무조건 쪼개면 JOIN이 많아져 성능이 저하되므로, 필요시 '반정규화'를 하기도 합니다.

i. ERD (Entity-Relationship Diagram)

데이터베이스의 뼈대(설계도)를 시각적으로 표현한 다이어그램입니다.

  • 1:1 관계: 사람 - 주민등록증
  • 1:N 관계: 부서(1) - 직원(N) (가장 흔한 구조)
  • N:M 관계: 학생(N) - 수강과목(M) (보통 중간에 매핑 테이블을 둬서 1:N 2개로 풉니다)

3. NoSQL (비관계형 DB)

RDBMS의 한계(유연성 부족, 확장 비용)를 극복하기 위해 등장한 DB입니다.

a. 문서형 (Document)

데이터를 JSON 형태로 저장합니다. 스키마가 자유롭습니다.


{
  "이름": "홍길동",
  "취미": ["독서", "게임"]
}
  • 대표 DB: MongoDB

b. Key-Value & 와이드 컬럼 & 그래프

  • Key-Value: '학번:1234' -> '홍길동' 처럼 단순한 키-값 구조. 속도 최상. (Redis)
  • 와이드 컬럼: 행마다 컬럼 개수와 종류가 달라도 됨. 대량 데이터 분산 처리에 특화. (Cassandra)
  • 그래프: 데이터 자체보다 '관계' 탐색 시 유리 (예: SNS 친구 추천). (Neo4j)

4. 특수 목적 DB

a. 시계열 DB & 인메모리 DB

  • 시계열 DB: 온도 센서, 주식 차트 등 '시간'에 따라 끊임없이 쏟아지는 데이터를 다룰 때 씁니다. (예: InfluxDB)
  • 인메모리 DB: 디스크가 아닌 RAM에 저장해 압도적으로 빠릅니다. (예: Redis) 주로 실시간 캐싱(Caching)에 사용됩니다.

b. 벡터 DB (Vector DB)

LLM(대형 언어 모델) 등 AI 시대의 핵심입니다. 비정형 데이터(텍스트, 이미지)를 숫자 배열(벡터)로 저장하고, 유사도를 순식간에 계산해 검색합니다.

5. 클라우드 DB란?

AWS, GCP, Azure에서 제공하는 "빌려 쓰는 데이터베이스(DBaaS)" 입니다.

클라우드 DB의 장점

  • 관리 자동화: 서버 설치, 백업, 보안 패치, 고장 시 복구를 알아서 해줍니다.
  • 탄력적 확장성: 트래픽이 폭주하면 클릭 몇 번으로 DB 성능을 즉시 늘릴 수 있습니다.
  • 대표 서비스: Amazon RDS, Google Cloud SQL

결론

세상에 모든 걸 완벽하게 처리하는 '만능 DB'는 없습니다.

✔️ 무결성과 복잡한 관계가 중요하면 RDBMS

✔️ 유연한 스키마와 대용량 확장이 필요하면 NoSQL

✔️ AI 검색이나 실시간 반응이 필요하면 특화 DB

목적에 맞게 여러 DB를 조합해서 쓰는 것이 핵심입니다!
